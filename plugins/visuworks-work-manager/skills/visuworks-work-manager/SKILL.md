# 비쥬웍스 작업 컨텍스트 업데이트 (쓰기 전용)

이 스킬은 Notion 정본을 **갱신**하는 절차다. 읽어오는 일(항상 참조)은 프로젝트
커스텀 지시문(또는 CLAUDE.md)이 처리하므로 여기서는 다루지 않는다.

## 4개 레이어 (무엇이 어디로 가는가)

| 레이어 | 담는 것 | 로드 | 갱신 방식 |
|---|---|---|---|
| **claude.md** | 규칙·표준·금지·용어 고정 | 항상 로드 | rewrite, 아주 짧게 |
| **context.md** | 지금 하는 일·활성 결정·열린 이슈 + **지식 문서 인덱스** | 항상 로드 | rewrite, 바운드 유지 |
| **지식 문서(주제별)** | 서브시스템·주제에 대한 정리된 이해 | on-demand | rewrite(살아있는 정리본) |
| **history.md** | "언제 무슨 대화에서 뭘 결정했나" 시간순 | on-demand | append-only |

핵심 구분:
- **context.md** = 휘발성·현재형·바운드. "지금 상황이 뭐야?"에 답한다.
- **지식 문서** = durable·정리된 이해·주제 단위. "X는 어떻게 되어 있지?"에 답한다.
- 판별 휴리스틱: **3개월 뒤에도 (지금 뭘 하든 무관하게) 참인 정리된 내용 → 지식 문서.**
  "이번 스프린트/이번 달 상태" → context.md. "YYYY-MM-DD에 …했다" → history.md.

## 대상 Notion 정본

아래 URL은 설치 시 입력받은 값(userConfig)을 사용한다. 메모리·추측 금지.

- claude.md  : ${user_config.claude_md_url}
- context.md : ${user_config.context_md_url}
- history.md : ${user_config.history_md_url}
- 지식 문서 부모 : ${user_config.knowledge_parent_url}
  — 주제별 지식 문서가 생성·보관되는 위치. 각 문서의 URL은 **context.md의 "지식 문서
  인덱스" 섹션에 등록**되며, 그 인덱스가 지식 문서 URL의 유일한 권위 있는 출처다.

## 접근 범위 규율 (워크스페이스 검색 방지)

- 접근 가능한 URL = 위 정본 URL + **context.md 인덱스에 등록된 지식 문서 URL**.
- 지식 문서를 찾을 때는 반드시 **context.md 인덱스에서 URL을 조회해 직접 fetch**한다.
  `notion-search` 등 워크스페이스 전체 검색으로 지식 문서를 "찾아다니지" 않는다.
- 워크스페이스 검색/외부 페이지 반입은 **자동으로 하지 않는다.** 아래 B 상태에서
  사용자 인가를 받은 경우에만 허용된다.
  (Cowork에서는 이 규칙이 플러그인 PreToolUse 훅으로도 강제되어, 워크스페이스 검색
  도구를 호출하면 사용자 승인 프롬프트가 뜬다. chat에서는 이 지시만 적용된다.)

## 페이지 상태별 처리 (모든 대상 페이지에 적용)

지정 URL을 fetch한 뒤 세 상태로 분기한다.

### A. 페이지가 없음 (미존재 / URL 무효)
- 해당 페이지를 **새로 생성한다.** (지식 문서라면 지식 문서 부모 아래에 생성)
- 생성 위치를 알 수 없으면 어디에 만들지 한 번 확인한다.
- 지식 문서를 새로 만들면 **context.md 인덱스에 그 링크를 등록**한다.
- 생성 후 계속 진행한다.

### B. 페이지는 있으나 비어 있음
- **자동으로 워크스페이스를 탐색하지 않는다.**
- 먼저 "○○(파일/문서)가 현재 비어 있음"을 사용자에게 명확히 알린다.
- 요청 처리에 다른 곳의 데이터가 필요하면, 그 사실과 함께 **"다른 곳에서 데이터를
  가져올까요?"라고 인가를 요청**한다. 명시적 승인이 있을 때만 탐색/반입한다.
- 단, 지금 작업이 **쓰기**라면 빈 페이지/문서에 첫 내용을 채우는 것은 정상 진행이다
  (외부 탐색이 아니므로 인가 불필요).

### C. 페이지가 있고 내용도 있음
- 정상 진행한다.

## 라우팅: 받은 내용을 어느 레이어로

저장 요청을 받으면 아래 순서로 분류한다.

1. **항상 지켜야 할 규칙·표준·금지·용어 고정인가?** → claude.md
   (예: 데이터 처리 규약, 명명 규칙, 재현성/일관성 원칙, 선호 스택 표기)
2. **특정 서브시스템·주제에 대한 정리된 이해인가? (3개월 뒤에도 유효)** → 해당 **지식 문서**
   (예: 주제/서브시스템별 문서 — subsystem-a, pipeline-x … 없으면 생성 + 인덱스 등록)
3. **지금 진행 중인 상태·활성 결정·열린 이슈인가?** → context.md
   (예: 진행 중 설계 단계, 핵심 정의 확정, 미해결 질문)
4. **"언제 무슨 대화에서 뭘 결정/다음 액션"인가?** → history.md

애매하면 어느 레이어인지 사용자에게 한 번 확인한다. 한 대화에서 여러 레이어에 걸치는
내용이 나오면 각 조각을 해당 레이어로 나눠 보낸다(단, 같은 내용을 중복 기록하지 않음).

## 레이어별 절차

### claude.md
- 규칙·표준 성격만. 매우 짧게 유지(항상 로드되므로).
- 상충하는 옛 규칙은 추가하지 말고 교체(rewrite).

### context.md
- "지금 하는 일 / 활성 결정 / 열린 이슈"와 "지식 문서 인덱스"를 유지한다.
- **바운드 유지**: 오래되거나 정착된 내용은 지우거나 지식 문서로 옮긴다.
- **Promotion**: 활성 결정이 확정·안정화되면 context의 활성 목록에서 빼고, 해당 지식
  문서에 정리해 반영한다(그리고 필요 시 history.md에 결정 사실을 한 줄 남긴다).
- 지식 문서 인덱스 항목 형식: `- <주제명> : <URL> — <1줄 설명>`

### 지식 문서 (주제별, on-demand)
- **주제 단위로 하나.** 없으면 부모 아래 생성하고 context.md 인덱스에 등록한다.
- **갱신은 append가 아니라 rewrite** — 최신 이해로 해당 섹션을 다시 쓴다(살아있는 정리본).
  상충하는 옛 서술은 남기지 말고 교체한다.
- 문서 상단에 `최종 갱신: YYYY-MM-DD`만 두고, 본문에는 개별 날짜를 흩뿌리지 않는다.
- 원문(대화 raw) 금지 — 정리·종합된 이해만.
- 문서가 너무 커지면(한 주제인데 500줄 초과 등) 하위 주제로 분할을 제안한다.

### history.md (대화 기준 / 정기 정리)
- 트리거: "이번 대화 정리", "history 업데이트", 정기 정리 요청.
- 형식(항목당 5줄 이내):
  ```
  ## YYYY-MM-DD — <한 줄 제목>
  - 다룬 주제:
  - 내린 결정:
  - 다음 액션:
  ```
- 최신이 위로 오게 append.
- 길어지면 오래된 항목을 별도 아카이브 페이지로 옮길지 제안(정본은 가볍게).

## 공통 쓰기 규칙

1. **쓰기 전 diff 요약 + 승인.** 어느 페이지/문서의 어디에 무엇을 추가·수정·교체할지
   짧게 보여주고, 승인 전에는 Notion에 쓰지 않는다. (지식 문서 rewrite는 before/after 요지를 보인다.)
2. **중복 방지.** 쓰기 전 대상 페이지를 fetch해 현재 상태를 확인하고, 겹치면 새로 추가하지
   말고 기존 항목을 갱신한다. 한 변경을 여러 레이어에 중복 기록하지 않는다.
3. **정밀 수정.** 해당 부분만 바꾸고 전체를 덮어쓰지 않는다.
4. 원문을 그대로 붙여넣지 않고 항상 요약·정리한다.

## 하지 말 것
- 승인 없이 Notion에 쓰기.
- 같은 내용을 여러 레이어에 중복 기록.
- context.md에 "전체 작업 정리"를 통째로 담아 always-load를 뚱뚱하게 만들기
  (그건 지식 문서로 보낸다).
- 지식 문서를 append로 무한 성장시키기(rewrite로 유지).
- history.md를 "항상 참조" 대상으로 끌어오기.
- 빈 페이지에서 **사용자 인가 없이 자동으로** 워크스페이스 검색/외부 반입(위 B 참고).
