# Obsidian vault 작업 컨텍스트 관리 (메인 정본)

비쥬웍스 작업 컨텍스트의 **메인 정본은 Obsidian vault**다. Notion은 회의록·팀 공용
문서 작성과 정보 읽기에만 쓴다(그쪽 쓰기는 visuworks-work-manager 스킬 담당).
이 스킬은 vault에 대한 읽기·검색·쓰기를 모두 담당한다.

## 발동·기록 사이클 (세션 지속)

이 스킬의 생애주기는 "명시 호출 → 세션 지속 감시 → 작업 단위 완료 시 승인 후 기록"이다.

1. **최초 발동은 명시 호출만.** 사용자가 스킬을 직접 지칭해 호출한 경우에만 활성화된다.
   호출 전에는 완전히 침묵한다 — 첨부 파일, 외부 소통 언급, "기록해줘"류 발언이 있어도
   발동하지 않는다.
2. **호출 후에는 세션이 끝날 때까지 지속 활성.** 재호출 불필요. 두 경로로 동작한다:
   - **사용자 요청 대응**: 기록·정리·저장 의도의 요청이면 문구가 정확히 일치하지 않아도
     (예: "이거 남겨놔", "아까 결정한 거 정리", "vault에 넣어줘") 대응한다.
   - **자동 판단**: 매 턴 종료 시 기록 필요 여부를 스스로 판단한다.
3. **기록 필요 판단 기준** — 아래 둘을 모두 만족할 때만 기록을 제안한다:
   - 레이어 라우팅에 걸리는 내용(결정·지식·상태 변화·첨부·외부 소통)이 있고,
   - **해당 작업 단위가 완료**되었다(산출물이 나왔거나 결정이 확정됨).
     작업이 진행 중이면 조용히 넘어간다 — 턴마다 조각 기록을 만들지 않는다.
4. **승인 후 쓰기.** 기록이 필요하면 무엇을 어느 파일에 작성/수정할지 요약을 제시하고
   **사용자 승인을 받은 뒤** 쓴다. 승인은 **작업 단위당 1회, 일괄**로 받는다 —
   파일별로 따로 묻지 않고 "history 1건 추가 + knowledge 2개 rewrite + index 갱신"처럼
   묶어서 제시한다. claude.md 수정이 포함되면 해당 diff를 승인 요약 안에 명시한다.
   기록할 것이 없으면 아무 말도 하지 않는다(보고 불필요).
5. **작업이 사용자 입력 대기로 멈추는 턴에서는** 진행이 장시간 끊길 수 있으므로,
   그 시점까지의 중간 상태를 기록해둘지 승인 요청을 함께 보낸다. 이때 중간 기록은
   **프로젝트 context.md의 열린 이슈에만** 남긴다 — 미완성 이해를 knowledge/·history에
   넣지 않는다. 작업 재개 후 완료되면 정식 기록으로 정리하고 열린 이슈에서 지운다.

## vault 경로

- vault 루트: `${user_config.obsidian_vault_path}` (설치 시 각자 입력)
- 파일 접근은 Read/Write/Edit/Grep 도구로 직접 한다. 경로에 접근할 수 없으면
  사용자에게 해당 폴더를 작업 폴더로 연결해 달라고 요청한다(추측으로 다른 경로를
  뒤지지 않는다).

## vault 구조 (없으면 생성)

```
<vault>/
  claude.md                 전역 규칙·표준 (하나만, rewrite, 아주 짧게)
  context.md                전역 인덱스: 활성 프로젝트 목록 + 각 프로젝트 링크
  knowledge/                프로젝트 무관 공용 지식 (툴·인프라·팀 공통)
  assets/                   프로젝트 무소속 첨부 원본
  projects/
    <프로젝트명>/
      context.md            프로젝트 현재 상태·활성 결정·열린 이슈
      history.md            프로젝트 이력 (append-only, 최신이 위)
      index.md              자산 인덱스: knowledge·assets·communications 목록 + 1줄 설명
      knowledge/            프로젝트 지식 문서 (주제별 1개, 첨부 정리본 포함)
      assets/               첨부 원본 보관 (편집 금지)
      communications/       외부 업체·부서 소통 기록 (주제별 1개, append)
      archive/              지식 문서 rewrite 전 스냅샷 (편집 금지)
```

- 처음 쓰는 파일/폴더는 승인된 기록 작업 안에서 생성한다. 단
  assets/·communications/·archive/·index.md는 **필요해진 시점에** 생성한다(빈 폴더 선제 생성 금지).
- 새 프로젝트 폴더를 만들면 **전역 context.md의 활성 프로젝트 목록에 등록**한다.

## 레이어 라우팅 (무엇이 어디로)

1. **항상 지켜야 할 규칙·표준·금지·용어 고정** → `claude.md` (전역, rewrite)
2. **정리된 이해, 3개월 뒤에도 유효** → 지식 문서
   - 특정 프로젝트에 종속 → `projects/<p>/knowledge/<주제>.md`
   - 프로젝트 무관(툴·인프라·팀 공통) → `knowledge/<주제>.md`
3. **지금 진행 중인 상태·활성 결정·열린 이슈** → `projects/<p>/context.md` (rewrite, 바운드 유지)
4. **"언제 무슨 대화에서 뭘 결정/다음 액션"** → `projects/<p>/history.md` (append)
5. **대화에 첨부된 파일 (원본 + 내용)** → `projects/<p>/assets/` + 지식 문서 (아래 "첨부 파일 처리")
6. **외부 업체·부서와 주고받은 내용·전달 파일** → `projects/<p>/communications/<주제>.md` (append)

프로젝트는 현재 작업 맥락에서 추론하고, 애매하면 한 번 확인한다.
한 대화에서 여러 레이어에 걸치면 조각별로 나눠 보내되 같은 내용을 중복 기록하지 않는다.

## 첨부 파일 처리 (원본 보관 + 자료화)

스킬 활성 상태에서 대화에 파일이 첨부되면 이 절차를 적용한다. 첨부 목적의 작업
(질문·분석 등)이 끝난 뒤, 작업 단위 일괄 승인에 보관·자료화 내용을 포함해 제안한다.

1. **원본 복사**: `projects/<p>/assets/YYYY-MM-DD_<원본파일명>`으로 복사한다
   (파일명 공백은 `-`로). assets/의 원본은 편집하지 않는다.
2. **버전 관리**: 기존 파일을 덮어쓰지 않는다. 새 버전은 새 날짜 접두사로 저장하되,
   같은 날 같은 이름이 다시 오면 `_v2` 접미사를 붙인다. 새 버전 저장 시
   지식 문서·index.md의 링크를 최신본으로 갱신하고 구본은 index에 "(구버전)"으로 표시한다.
3. **자료화(정리본)**: 첨부 내용을 요약·정리해 `knowledge/<주제>.md`에 반영한다 —
   기존 주제 문서가 있으면 rewrite로 통합, 없으면 신설. 첨부 1개 = 문서 1개가 아니라
   **주제 단위**로 묶는다. frontmatter에 `source: "[[YYYY-MM-DD_<파일명>.<ext>]]"` 필드를
   추가하고 본문 첫머리에도 원본 링크를 둔다.
4. **링크 표기**: 비마크다운 파일의 위키링크는 **확장자를 반드시 포함**한다 —
   `[[2026-07-14_견적서.pdf|견적서 원본]]`. (확장자 없는 위키링크는 .md로 해석되므로 금지.)
5. **프로젝트 매핑**: 현재 맥락에서 추론하고 애매하면 한 번 확인. 어떤 프로젝트에도
   속하지 않으면 전역 `knowledge/` + 전역 `assets/`에 저장한다. 전역 knowledge/assets는
   index 대상이 아니다(index.md는 프로젝트 전용).
6. 저장 후 **index.md를 같은 턴에 갱신**하고 결과를 요약 보고한다.

## 외부 소통 기록 (communications/)

스킬 활성 상태에서 외부 업체·부서와 주고받은 내용이나 파일 전달이 언급되면,
작업 단위 일괄 승인에 소통 기록을 포함해 제안한다.

- 파일 단위 = **업무 주제**(상대 단위가 아님): `communications/<주제>.md`
- frontmatter: `updated`, `project`, `counterparts: [<업체/부서>]`, `tags`
- **append-only, 최신이 위.** 항목 형식:
  ```
  ## YYYY-MM-DD — <상대(업체/부서)> — <한 줄 요약>
  - 방향: 발신 | 수신
  - 전달 파일: [[YYYY-MM-DD_<파일명>.<ext>]] (없으면 줄 생략)
  - 내용: 오간 내용 2~3줄 요약
  - 후속: 다음 액션·대기 사항 (없으면 생략)
  ```
- **history.md와의 분담**: communications = 외부 상대와 "무엇을 주고받았나"의 교신 원장,
  history = 프로젝트 내부 "무엇을 결정했나"의 이력. 외부 소통이 결정·상태에 영향을 주면
  history에 한 줄 + `[[communications/<주제>]]` 링크만 남기고 상세를 중복 기록하지 않는다.
- 전달한 파일이 vault에 없으면 먼저 assets/에 보관(첨부 파일 처리 절차)한 뒤 링크한다.

## index.md (프로젝트 자산 인덱스)

- 역할: 해당 프로젝트 knowledge/·assets/·communications/의 **전체 파일 목록 + 1줄 설명**.
  context.md·history.md·archive/는 목록 대상이 아니다.
- **갱신 시점**: 세 폴더에 파일을 생성·개명·삭제하거나 설명이 달라지면
  **같은 작업 턴 안에서 즉시** rewrite로 갱신한다(별도 승인 불필요 — 원 작업 승인에 포함된 것으로 본다).
- 프로젝트 context.md에는 `- 자산 인덱스: [[index]]` 한 줄만 둔다. 기존 프로젝트에
  옛 "지식 문서 인덱스" 섹션이 남아 있으면 처음 만질 때 index.md로 **1회 이관**한다.
- 템플릿:
  ```
  ---
  updated: YYYY-MM-DD
  project: <프로젝트명>
  ---
  # <프로젝트명> 인덱스

  ## Knowledge
  - [[<주제>]] — <1줄 설명>

  ## Assets
  - [[YYYY-MM-DD_<파일명>.<ext>]] — <1줄 설명> (관련: [[<주제>]])

  ## Communications
  - [[communications/<주제>]] — <상대> · <1줄 설명>
  ```
- 한 소통 주제에 여러 상대가 섞이면 index에는 대표 상대 + `외`로 표기한다.

## 지식 문서 스냅샷 (archive/)

- 지식 문서의 **기존 섹션을 삭제·대체하는 큰 rewrite 전**에는 수정 전 상태를
  `projects/<p>/archive/YYYY-MM-DD_<문서명>.md`로 복사한 뒤 진행한다.
  (단순 추가·frontmatter 갱신·오탈자 수정은 스냅샷 불필요.)
- 같은 날 중복은 `_v2`. archive 사본은 편집·링크 갱신 대상이 아니며(그 시점 그대로 보존),
  index.md에 목록화하지 않는다.
- assets 원본은 불변, communications·history는 append-only이므로 스냅샷 대상이 아니다.

## 프로젝트 범위 읽기 제한

프로젝트가 특정된 대화에서 vault 읽기·검색(Grep 포함)은 다음으로 한정한다:

1. 전역 `claude.md`·`context.md` — 항상 허용
2. `projects/<p>/` 전체 — 허용. 단 **index.md를 먼저 읽고** 필요한 문서만 골라 읽는다(전량 로드 금지)
3. 전역 `knowledge/` — 프로젝트 무관 공용 지식이 필요할 때만 on-demand

다른 프로젝트 폴더(`projects/<q>/`)는 **사용자가 명시적으로 요청·언급했을 때만** 읽는다.
vault 전체 Grep이 필요하면 path를 `projects/<p>/`로 한정해 실행한다.

## Obsidian 네이티브 규칙

- **frontmatter**: 지식 문서 상단에 YAML로 유지한다.
  ```yaml
  ---
  updated: YYYY-MM-DD
  project: <프로젝트명 | common>
  tags: [<주제 태그>]
  ---
  ```
- **위키링크**: 문서 간 참조는 `[[문서명]]`으로 연결한다. 지식 문서끼리, context.md →
  지식 문서, history 항목 → 관련 지식 문서 링크를 적극 사용한다.
  비마크다운 파일은 확장자 포함 필수(`[[파일.pdf]]`).
- **인덱스**: 프로젝트 index.md의 줄 형식은 위 "index.md" 섹션의 템플릿이 정본이다
  (기본형 `- [[<파일>]] — <1줄 설명>`, Assets는 `(관련: [[<주제>]])`, Communications는 `<상대> ·` 추가).
- 파일명 = 주제명(한글 가능, 공백 대신 `-` 권장). 위키링크가 파일명 기준이므로
  파일명 변경 시 링크를 함께 갱신한다(assets 파일 개명 시에도 동일).

## 레이어별 갱신 방식

- **claude.md**: 규칙만, 매우 짧게. 상충하는 옛 규칙은 교체. 수정 diff를 일괄 승인
  요약 안에 반드시 명시한다.
- **context.md (전역/프로젝트)**: rewrite. 오래되거나 정착된 내용은 지식 문서로 옮기고
  지운다(바운드 유지). 확정된 활성 결정은 지식 문서로 promotion + history에 한 줄.
- **지식 문서**: append가 아니라 **rewrite** — 최신 이해로 해당 섹션을 다시 쓴다.
  기존 섹션을 삭제·대체하는 큰 rewrite 전에는 archive/ 스냅샷을 먼저 남긴다.
  원문(대화 raw) 금지, 정리된 이해만. frontmatter의 `updated`만 갱신하고 본문에
  날짜를 흩뿌리지 않는다. 500줄 초과 시 하위 주제 분할 제안.
- **history.md**: append-only, 최신이 위. 항목당 5줄 이내:
  ```
  ## YYYY-MM-DD — <한 줄 제목>
  - 다룬 주제:
  - 내린 결정:
  - 다음 액션:
  ```
- **index.md**: rewrite. knowledge/assets/communications 파일 변동 시 같은 턴에 즉시.
- **communications/<주제>.md**: append-only, 최신이 위.

## 쓰기 규칙

1. **승인 후 쓴다.** 어떤 vault 쓰기도 사전 승인 없이 실행하지 않는다. 작업 단위가
   완료되면 기록할 내용 전체(대상 파일·작성/수정 요지·claude.md diff 포함)를 요약해
   **한 번에** 승인 요청하고, 승인 후 일괄 실행 → 결과를 간단히 보고한다.
   같은 작업 단위 안의 후속 정정(오탈자, 링크 수정)은 재승인 불필요.
2. **중복 방지**: 쓰기 전 대상 파일을 읽어 현재 상태를 확인하고, 겹치면 기존 항목을 갱신한다.
3. **정밀 수정**: 해당 부분만 바꾸고 파일 전체를 덮어쓰지 않는다(Edit 우선).
   단, rewrite로 지정된 파일(context.md·지식 문서·index.md)은 바운드 유지를 위해
   전체 재작성이 정상이다 — Edit 우선은 append 파일(history·communications)과 부분 정정에 적용.
4. 원문을 그대로 붙여넣지 않고 항상 요약·정리한다.

## Notion과의 분담

- **이 스킬(Obsidian)**: 정본 기록소. 규칙·결정·지식·상태·이력·첨부·소통 기록의 모든 저장.
  Notion에는 claude.md/context.md/knowledge/history 레이어가 **없다.**
- **visuworks-work-manager(Notion)**: 회의록 작성, 팀 공용 문서 작성, "Notion에" 명시 요청.
- Notion에서 정보를 **읽어와** vault에 정리하는 작업은 이 스킬이 담당한다.
  단, 읽기는 사용자가 지정한 페이지에 한하며 워크스페이스 검색/쿼리는 사용자
  인가 없이 실행하지 않는다.

## 하지 말 것

- 명시 호출 전에 발동하거나 기록을 제안하기.
- 승인 없이 vault에 쓰기 (일괄 승인 원칙 — 파일별 분할 승인 요청도 금지).
- 작업 단위가 끝나지 않았는데 기록 제안하기 (입력 대기로 멈추는 시점의 중간 기록 제안은 예외).
- 중간 기록을 context.md 열린 이슈 밖(knowledge/·history)에 남기기.
- 같은 내용을 여러 레이어/파일에 중복 기록.
- 전역 context.md에 프로젝트 상세를 담기(프로젝트 context.md로 보낸다).
- 지식 문서를 append로 무한 성장시키기(rewrite로 유지).
- vault 밖 경로에 쓰기, 또는 접근 불가 시 경로 추측.
- assets/ 원본·archive/ 사본을 편집하거나, assets/에 마크다운 정리본을 두기(정리본은 knowledge/).
- index.md 갱신 없이 knowledge/assets/communications에 파일을 추가·개명·삭제.
- 외부 소통 상세를 history.md에 중복 기록.
- 프로젝트가 특정된 상태에서 다른 프로젝트 폴더를 임의로 읽기.
- 확장자 없는 비마크다운 위키링크(`[[파일]]` → `[[파일.pdf]]`).
- 지식 문서의 큰 rewrite를 archive/ 스냅샷 없이 진행.
